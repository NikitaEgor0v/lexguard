# Краевые Случаи и Риски Архитектуры (Edge Cases & Risks)

В ходе аудита кодовой базы были выявлены потенциально "хрупкие" места, которые могут стать причиной ошибок или ухудшения пользовательского опыта. Ниже перечислены основные краевые случаи и технические компромиссы текущей архитектуры LexGuard.

## 1. Лимиты на размер документа
**Где:** `api/routes.py`
**Суть:** В целях предотвращения Out-of-Memory (OOM) ошибок, установлен жесткий лимит `MAX_SEGMENTS_PER_DOCUMENT = 80`. Если препроцессор разбивает документ более чем на 80 частей, API возвращает ошибку `422`. 
**Риск:** Крупные контракты (особенно многостраничные приложения и технические задания) могут быть отклонены системой.

## 2. Определение типа контракта по началу текста
**Где:** `services/analyzer.py` -> `_classify_contract_type()`
**Суть:** Для определения типа договора LLM анализирует только первые один или два сегмента (до 800 символов). 
**Риск:** В российских реалиях преамбула (указание сторон, оснований действий) может быть очень длинной. Если слово "Договор подряда" находится за пределами первых 800 символов, система неверно классифицирует договор как `"иной"`, что приведет к снижению релевантности RAG-поиска (будут искаться только универсальные нормы).

## 3. Зависание UI при "падении" Celery-таски
**Где:** Взаимодействие `routes.py`, `analyzer.py` и Redis.
**Суть:** Фронтенд опрашивает статус `/analyze/{id}`. Бэкенд читает статус из Redis (`progress:{id}`). Если Celery worker внезапно падает (например, OOM или краш контейнера) в середине процесса, ключ в Redis просто протухнет (установлен TTL 3600 секунд). 
**Риск:** Фронтенд может "зависнуть" в бесконечном поллинге или показывать устаревший статус, так как нет явного механизма передачи статуса `failed` из Celery в БД, если worker был убит системой (SIGKILL).

## 4. Скрытый (Silent) Fallback парсера JSON
**Где:** `services/analyzer.py` -> `_parse()`
**Суть:** Если LLM галлюцинирует и возвращает невалидный JSON, метод `_parse()` перехватывает исключение и возвращает захардкоженный ответ: `is_risky=True`, `risk_level=LOW`, `recommendation="Проверьте вручную"`.
**Риск:** Настоящие ошибки генерации (или пустые ответы) скрываются от разработчика. Вместо того, чтобы логировать критическую проблему или пере-генерировать ответ, система маскирует ошибку под "слабый риск". При массовой галлюцинации документ будет усеян "низкими" рисками без реального анализа.

## 5. Тихие ошибки Redis
**Где:** `services/analyzer.py` -> `analyze()`
**Суть:** При обновлении прогресса анализ ловит `Exception` при работе с Redis и игнорирует его (`pass`). 
**Риск:** Это повышает аптайм (анализ не падает, если упал Redis), но при этом ломается UX на клиенте — прогресс-бар просто замирает на 0% до самого конца анализа.

## Рекомендации по устранению
1. Внедрить механизм обработки "мертвых" Celery-задач с таймаутом, чтобы переводить анализ в статус `error`.
2. Увеличить контекст для классификации типа документа или искать ключевые слова (regex) по всему документу перед вызовом LLM.
3. Добавить счетчик "сломанных JSON" на один документ. Если LLM ошибается в >20% сегментов, прерывать анализ и выдавать ошибку генерации.

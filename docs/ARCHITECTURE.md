# Техническая архитектура и документация проекта LexGuard

## 1. Обзор системы и Архитектура

**LexGuard** — это интеллектуальная система автоматизированного юридического анализа контрактов. Проект предназначен для выявления юридических, финансовых и операционных рисков в текстах договоров (PDF, DOCX) с использованием больших языковых моделей (LLM) и технологии поисковой генерации (RAG).

Система построена на принципах строгой микросервисной архитектуры и оркестрируется с помощью Docker Compose, что обеспечивает легкость масштабирования и отказоустойчивость.

### Разбиение на микросервисы
1. **Frontend (Nginx / Vanilla JS)**: Отвечает за реактивный пользовательский интерфейс. Взаимодействует с бэкендом через REST API, использует механизм long-polling (или периодический polling к Redis) для потокового отображения прогресса анализа без перезагрузок страницы.
2. **Backend (FastAPI)**: Ядро системы. Выполняет обработку HTTP-запросов, аутентификацию (JWT), управление базой данных и маршрутизацию задач в очередь Celery.
3. **Celery Worker & Redis**: Асинхронный контур. Анализ документов требует длительного времени, поэтому FastAPI мгновенно возвращает `analysis_id`, а Celery в фоновом режиме порционно обрабатывает документ. Redis используется как брокер сообщений для Celery, а также как in-memory хранилище для отслеживания прогресса (прогресс обновляется по ключу `progress:{analysis_id}`).
4. **Relational Database (PostgreSQL)**: Хранилище бизнес-логики. Здесь хранятся учетные данные пользователей, история анализов, риск-отчеты (структурированные результаты `analysis_results` и `risk_items`) и история чат-сессий (по принципам Get-or-Create). Управление миграциями осуществляется через Alembic.
5. **Vector Search (Qdrant)**: Векторная база данных. Хранит математические эмбеддинги системных правовых норм и кастомных эталонов пользователей для поиска ближних соседей (KNN).
6. **LLM Engine (Ollama)**: Контейнер для запуска нейросетей семейства Gemma. По умолчанию используется модель, заданная в переменной `LLM_MODEL` (например, `gemma2:2b`). Изолирован от основного бэкенда для обеспечения безопасности памяти и квотирования ресурсов.

### Поток данных (Data Flow)
Краткий жизненный цикл обработки документа:
`Upload → FastAPI → Preprocessor (разбиение на сегменты) → Celery Queue → Redis (статус) → Qdrant Vector Search (для каждого сегмента) → Ollama RAG Generation → JSON Parse → PostgreSQL Save → Frontend`

Подробнее о потоках данных можно прочитать в [API_AND_DATA_FLOW.md](API_AND_DATA_FLOW.md).

## 2. ML-Пайплайн и RAG

Контур машинного обучения в LexGuard заточен на отказоустойчивость и точность контекста.

### Стратегия сегментации (Zero-Data-Loss)
Разбиение юридических текстов требует хирургической точности. В `PreprocessorService` реализован алгоритм "безпотерьной" буферизации.
- Документ бьется по нумерации параграфов или двойным абзацам.
- Короткие пункты накапливаются в буфере до достижения оптимального размера (около 400 символов, максимум до 800).
- Существует жесткое ограничение на количество сегментов (до 80), чтобы избежать перегрузки.

### Векторизация и Qdrant Threshold
Модуль `RAGService` использует мультиязычный энкодер `intfloat/multilingual-e5-base` для перевода текста сегмента в 768-мерное пространство.
Поиск происходит с фильтрацией по типу документа (определяемому автоматически с помощью LLM перед RAG-поиском) с порогом отсечения `SCORE_THRESHOLD = 0.55`. Подробности логики RAG — в [RAG_AND_LOGIC.md](RAG_AND_LOGIC.md).

## 3. Схема Базы Данных

Адаптация системы построена на стэке **PostgreSQL + SQLAlchemy 2.0 + Alembic** для декларативного маппинга и управления миграциями.

* **Users (`users`)**: Управление доступом. Пароли хэшируются с применением `bcrypt`.
* **Documents (`user_documents`)**: Метаданные загруженных пользовательских эталонов для персонального RAG. Сами эмбеддинги лежат в Qdrant, но связь установлена через `document_id`.
* **Analysis Results (`analysis_results` & `risk_items`)**: Связь один-ко-многим. В таблице results хранится сводный скоринг (score, counts), а в risks — детальная разбивка каждого пункта с его уровнем (`high`, `medium`, `low`, `none`) и категорией (Финансовый, Операционный и т.д.).
* **Chat History (`chat_sessions` & `chat_messages`)**: Персистентная история. `ChatRepository.create_session` работает по паттерну `Get-or-Create`.

## 4. Обеспечение качества и Надежность (QA)

Архитектура системы включает ряд защитных механизмов:
* **Устойчивость к RAG-Галлюцинациям:** Встроенная логика fallback и экранирование промпта (системный промпт изолирован от пользовательского) не позволяет LLM выдумывать риски из воздуха.
* **Отказоустойчивость инфраструктуры:** Если контейнер Qdrant недоступен, FastAPI переключается на эвристический `_fallback()` механизм, основанный на словарях ключевых слов. Если Redis недоступен, прогресс просто игнорируется и сервис не "падает" при запросах.
* **Стойкий парсинг JSON:** Для защиты от нестабильной генерации моделей (Markdown слои ` ```json `, лишний текст), функция `_parse()` использует очистку строк. В случае фатальной ошибки парсинга, возвращается дефолтный риск уровня `low` для ручной проверки.

Об уязвимостях и хрупких местах системы подробнее описано в [EDGE_CASES_AND_RISKS.md](EDGE_CASES_AND_RISKS.md).

## 5. Инструкция по развертыванию

Проект полностью контейнеризирован. Вам потребуется установленный Docker Desktop.

1. **Базовый запуск инфраструктуры**
   ```bash
   docker compose up -d --build
   ```
   *Это поднимет Nginx, FastAPI, PostgreSQL, Redis, Celery, Qdrant и скачает движок Ollama.*

2. **Загрузка и монтирование весов LLM**
   Дождитесь завершения инициализации модели:
   ```bash
   docker logs -f lexguard-ollama-init
   ```

3. **Накатывание миграций базы данных**
   Инициализируйте схему в PostgreSQL через Alembic:
   ```bash
   docker exec -it lexguard-backend alembic upgrade head
   ```

4. **Проверка работы системы**
   Бэкенд доступен на порту `8000`. Панель управления Qdrant: `http://localhost:6333/dashboard`.

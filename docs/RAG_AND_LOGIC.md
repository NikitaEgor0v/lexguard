# RAG и Бизнес-логика

Модуль Legal Analyzer использует подход RAG (Retrieval-Augmented Generation) для предоставления LLM правового контекста перед принятием решения о наличии риска.

## 1. Векторизация и база данных (Qdrant)

За хранение и поиск эталонных правовых норм отвечает сервис `Qdrant`. 

- **Модель энкодера:** `intfloat/multilingual-e5-base`
- **Размерность вектора:** 768
- **Метрика дистанции:** Cosine Distance
- **Коллекция:** `legal_norms`

Все эталонные нормы (хранящиеся в `data/legal_norms.json`) преобразуются в векторы и помещаются в коллекцию Qdrant. Для каждой нормы сохраняются метаданные (payload): тип договора, категория риска, критичность, юридические паттерны и паттерны уловок.

## 2. Определение типа контракта

Перед началом покомпонентного RAG-анализа, система выполняет пред-анализ документа:
1. Берется превью договора: первый или два первых сегмента документа. Текст обрезается до 800 символов (`MAX_CLASSIFY_PREVIEW_CHARS`).
2. Превью отправляется в Ollama с промптом `CONTRACT_CLASSIFY_PROMPT`.
3. LLM определяет тип контракта (например, "аренда", "подряд", "нда").
4. Если LLM ошибается или не может определить, по умолчанию устанавливается тип `"иной"`.

## 3. RAG-поиск и фильтрация

Для каждого сегмента договора, который анализируется, происходит следующий алгоритм:
1. Сегмент переводится в вектор (`query: {текст}`).
2. Формируется объект фильтрации (`Filter` / `FieldCondition`), который требует совпадения поля `contract_type` с определенным типом контракта (с учетом алиасов) или универсальными типами (`все`, `all`).
3. Qdrant ищет топ-2 (`TOP_K = 2`) ближайших соседей.
4. Минимальный порог отсечения `SCORE_THRESHOLD = 0.55`. Все, что имеет сходство ниже — отбрасывается.
5. Параллельно, если пользователь загрузил свои кастомные документы (`user_id` передан), система производит поиск по пользовательским эталонам.

## 4. LLM Анализ (Ollama)

Найденные эталонные нормы (стандартные и пользовательские) конкатенируются и вставляются в промпт для LLM вместе с самим сегментом.

- **Системный промпт (`SYSTEM_PROMPT`):** Задает жесткие правила возврата ТОЛЬКО валидного JSON. Содержит точные критерии для классификации `risk_level` (high, medium, low, none) и `risk_category` (финансовый, правовой и т.д.).
- **Генерация:** Запрос отправляется в Ollama с параметрами конфигурации модели (задается через `get_model_config()`).
- **Фоллбек (Fallback):** Если Ollama недоступна, используется ручной метод на основе ключевых слов (словарный метод в `rag._fallback()`), который возвращает предупреждения по базовым категориям.

## 5. Группировка рисков

После анализа всего документа, сырой список рисков обрабатывается в `risk_grouping.py`:
- Риски со статусом `is_risky=False` игнорируются.
- Риски разбиваются по корзинам (buckets) в зависимости от их категории (`risk_category`).
- Внутри каждой категории ведется подсчет рисков уровня high, medium, low.
- Итоговые группы сортируются: в начале идут категории с наибольшим количеством критических (high) рисков, затем medium и т.д.
- Это позволяет фронтенду сразу отображать наиболее проблемные зоны контракта.

# API и Потоки Данных

Взаимодействие между Frontend-клиентом и Backend-сервером построено на REST API. В связи с длительным процессом анализа, используется асинхронный паттерн с фоновыми задачами и поллингом (polling).

## Основные эндпоинты

### `POST /api/v1/analyze`
**Назначение:** Загрузка документа (PDF/DOCX) и запуск анализа.
**Процесс:**
1. Проверка формата и размера файла (максимум 15 МБ).
2. Запуск препроцессора (`PreprocessorService`), который парсит текст и разбивает его на сегменты.
3. Валидация: Документ не должен быть пустым и не должен превышать 80 сегментов (`MAX_SEGMENTS_PER_DOCUMENT`).
4. Генерация `analysis_id` и отправка задачи в очередь Celery (`analyze_document_task.delay`).
5. **Ответ:** `202 Accepted` с `analysis_id` и статусом `processing`.

### `GET /api/v1/analyze/{analysis_id}`
**Назначение:** Получение статуса или результатов анализа.
**Процесс:**
1. Бэкенд проверяет наличие результата в PostgreSQL. Если анализ завершен, возвращается полная структура `AnalysisResponse`.
2. Если анализ еще идет, бэкенд обращается в Redis за ключом `progress:{analysis_id}`.
3. Формируется прогресс (например `45/80`) и отдается на фронтенд в виде `progress_percent` и текстового `progress_label`.
4. Если Redis недоступен, бэкенд не падает, а отдает дефолтное значение прогресса `0/1`.

### `GET /api/v1/analyze/{analysis_id}/grouped`
**Назначение:** Получение сгруппированных результатов для отображения в интерфейсе.
**Процесс:**
1. Загрузка данных анализа из БД.
2. Прогон данных через функцию `group_analysis_risks`.
3. Возврат структуры, где риски разбиты по категориям (`groups`), посчитано их количество и отсортировано по убыванию критичности.

### `GET /api/v1/analyses`
**Назначение:** Получение истории анализов пользователя (с пагинацией).

## Frontend Data Flow

Во фронтенде (`frontend/js`) работа с API обернута в центральный механизм в `api.js`.
- Функция `apiFetch` автоматически подставляет заголовки, обрабатывает 401 ошибки (Unauthorized) и вытягивает сообщения об ошибках из JSON.
- В файле `analysis.js` реализован поллинг: клиент делает запросы к `/api/v1/analyze/{id}` каждые пару секунд, обновляя UI прогресс-бара.
- Как только сервер возвращает `status: completed`, клиент вызывает `/analyze/{id}/grouped`, чтобы отрисовать результаты в разбивке по категориям рисков.
